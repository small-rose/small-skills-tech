# 技术问题解答 SOP

## 触发条件

当用户提出以下类型问题时触发：

| 问题类型 | 关键词示例 | 说明 |
|---------|-----------|------|
| 技术咨询 | "如何解决..."、"怎么配置..." | 一般性技术问题 |
| 错误排查 | "报错了..."、"运行失败..."、"不起作用..." | 错误诊断 |
| 环境配置 | "怎么安装..."、"如何配置..."、"环境变量..." | 环境搭建 |
| 代码调试 | "这段代码有问题..."、"调试..." | 代码问题 |
| 性能优化 | "太慢了..."、"优化..."、"性能..." | 性能问题 |
| 部署相关 | "部署..."、"上线..."、"发布..." | 部署问题 |

---

## 核心流程

### 快速响应路径（简单问题）

对于简单问题，直接回答，无需完整流程：

| 问题类型 | 判断标准 | 处理方式 |
|---------|---------|---------|
| 版本查询 | "XX 版本是多少" | 直接回答 |
| 语法问题 | 代码语法错误 | 直接给出正确写法 |
| 概念解释 | "什么是 XX" | 直接解释 |
| 快速命令 | "怎么用 XX 命令" | 直接给出命令 |

**快速响应示例**:
```
用户: Python 怎么查看版本？
AI: python --version
```

### 标准流程（复杂问题）

对于复杂问题，执行完整 SOP 流程：

### 阶段 1: 问题分析与分类

**目标**: 准确理解用户问题，识别技术栈

| 步骤 | 操作 | 输出 |
|------|------|------|
| 1.1 | 提取关键词 | 技术类型、组件、错误信息 |
| 1.2 | 识别技术大类 | Java/Python/Nginx/... |
| 1.3 | 识别技术小类 | Spring Boot/Django/... |
| 1.4 | 识别组件版本 | 如果未提供，主动询问 |

**检测规则**: 参考 `references/tech-stack-detection.md`

**版本检测**: 如果用户未提供版本信息，主动询问：

> "为了更准确地解答，请提供以下信息：
> - 操作系统类型和版本
> - [技术] 的版本（如 Java 版本、Python 版本）
> - 相关框架/组件的版本
> - 错误信息的完整内容"

---

### 阶段 2: 环境识别与确认

**目标**: 自动检测用户工程环境，确认技术栈详情

| 步骤 | 操作 | 输出 |
|------|------|------|
| 2.1 | 自动检测工程环境 | 操作系统、语言版本、框架版本 |
| 2.2 | 识别依赖版本 | pom.xml/package.json/... |
| 2.3 | 识别配置文件 | application.yml/nginx.conf/... |
| 2.4 | 信息不足时引导用户提供 | 环境信息收集完成 |

**自动检测方法**:

```bash
# Java 项目
java -version
mvn -version
cat pom.xml | grep -E "<version>|<parent>"

# Python 项目
python --version
pip --version
cat requirements.txt
cat pyproject.toml

# Nginx
nginx -v
cat /etc/nginx/nginx.conf

# Node.js
node --version
npm --version
cat package.json
```

**环境检查脚本**: 参考 `scripts/check-environment.ps1`

---

### 阶段 3: 资料搜集与验证

**目标**: 查找权威资料，验证来源可靠性

| 步骤 | 操作 | 输出 |
|------|------|------|
| 3.1 | 优先查阅官方文档 | 官方解决方案 |
| 3.2 | 搜索权威技术社区 | 社区解决方案 |
| 3.3 | 验证信息来源权威性 | 来源可信度评估 |
| 3.4 | 标注未验证信息 | 明确标注 |
| 3.5 | 禁止猜测 | 信息不足时主动询问 |

**来源权威性层级**:

| 层级 | 来源类型 | 可信度 | 示例 |
|------|---------|--------|------|
| 1 | 官方文档 | ⭐⭐⭐⭐⭐ | Spring 官网、Python 官网 |
| 2 | 官方 GitHub | ⭐⭐⭐⭐⭐ | spring-projects/spring-boot |
| 3 | 权威社区 | ⭐⭐⭐⭐ | Stack Overflow、GitHub Issues |
| 4 | 官方博客 | ⭐⭐⭐⭐ | Spring Blog、Python Blog |
| 5 | 技术书籍 | ⭐⭐⭐ | 出版社出版的书籍 |
| 6 | 个人博客 | ⭐⭐ | 需要验证 |
| 7 | AI 生成 | ⭐ | 必须验证 |

**链接验证**: 使用 `webfetch` 工具验证链接有效性

```markdown
**来源**: https://spring.io/projects/spring-boot
**验证状态**: ✅ 已验证（2026-06-18）
**可信度**: ⭐⭐⭐⭐⭐ 官方文档
```

或

```markdown
**来源**: https://example.com/article
**验证状态**: ❌ 无法验证
**失败原因**: 404 页面不存在
**可信度**: ⭐⭐ 社区博客（未验证）
```

**验证规范**: 参考 `references/source-verification.md`

### 阶段 4: 问题解答与引用

**目标**: 提供结构化解答，每个答案附带来源

| 步骤 | 操作 | 输出 |
|------|------|------|
| 4.1 | 提供结构化解答 | 问题分析 + 解决方案 |
| 4.2 | 每个答案附带来源引用 | 引用链接 + 可信度 |
| 4.3 | 区分官方 vs 社区方案 | 方案分类 |
| 4.4 | 标注未验证信息 | 警告标注 |
| 4.5 | 提供多种解决方案 | 备选方案 |

**回答模板**: 参考 `references/answer-template.md`

---

### 阶段 5: 验证与测试

**目标**: 提供验证方案，确认解决方案有效性

| 步骤 | 操作 | 输出 |
|------|------|------|
| 5.1 | 评估测试成本和复杂度 | 测试评估报告 |
| 5.2 | 询问用户测试环境 | 用户确认 |
| 5.3 | 编写验证案例 | 测试脚本 |
| 5.4 | 提供回归测试 | 测试用例 |

**测试复杂度评估**:

| 复杂度 | 说明 | 处理方式 |
|--------|------|---------|
| **低** | 简单命令、配置修改 | 直接提供测试脚本 |
| **中** | 需要特定环境、依赖 | 询问用户环境，提供测试步骤 |
| **高** | 需要复杂环境、数据 | 提供测试方案，让用户确认 |

**用户环境确认流程**:

```
评估测试复杂度
    ↓
├─ 低复杂度 → 直接提供测试脚本
└─ 中/高复杂度 → 询问用户
                 ↓
                 "此测试需要 [环境要求]，您的环境是否满足？"
                 ↓
                 ├─ 用户确认满足 → 提供详细测试步骤
                 └─ 用户确认不满足 → 提供简化测试方案
                                   或建议用户搭建环境
```

**测试用例指南**: 参考 `references/test-case-guidelines.md`

---

### 阶段 6: 文档化与反馈

**目标**: 记录解决方案，更新知识库

| 步骤 | 操作 | 输出 |
|------|------|------|
| 6.1 | 生成问题分析报告 | 分析文档 |
| 6.2 | 记录解决方案和验证结果 | 解决方案库 |
| 6.3 | 提供后续建议 | 优化建议 |
| 6.4 | 更新技术栈知识库 | 持续学习 |

**动态学习机制**:

| 触发条件 | 操作 |
|---------|------|
| 用户提问新技术栈 | 自动添加到 known-tech-stacks.json |
| 用户反馈解决方案有效 | 标记为已验证 |
| 用户提供新版本信息 | 更新版本列表 |
| 发现新的常见问题 | 添加到 common_issues |

## 来源引用规范

### 引用格式

```markdown
**来源**: [完整 URL]
**验证状态**: [✅ 已验证 / ❌ 未验证]
**验证时间**: [日期]
**可信度**: [⭐⭐⭐⭐⭐]
**来源类型**: [官方文档 / 社区讨论 / 博客文章 / ...]
```

### 未验证信息标注

```markdown
> ⚠️ **未经过验证**: 此解决方案来自社区讨论，尚未在官方文档中确认。
> 请在测试环境中验证后再应用到生产环境。
```

### 链接验证失败处理

| 失败原因 | 处理方式 |
|---------|---------|
| 404 页面不存在 | 说明链接已失效，提供替代方案 |
| 403 访问被拒绝 | 说明访问限制，提供摘要信息 |
| 连接超时 | 说明网络问题，建议用户自行验证 |
| DNS 错误 | 说明域名无法解析，提供已知信息 |

## 故障排查方法论

### 5 Whys 方法

通过连续追问"为什么"，深入挖掘根本原因。

### 鱼骨图分析

### 二分法排查

通过二分法定位问题范围，逐步缩小问题空间。

提出假设 → 设计验证 → 执行测试 → 确认/否定假设。

## 常见问题处理模式

### Java 问题

| 问题类型 | 常见原因 | 解决方案 |
|---------|---------|---------|
| 依赖冲突 | 版本不兼容 | 排除依赖、指定版本 |
| 配置错误 | 配置项缺失 | 检查配置文件 |
| 内存溢出 | 堆内存不足 | 调整 JVM 参数 |

### Python 问题

| 问题类型 | 常见原因 | 解决方案 |
|---------|---------|---------|
| 依赖冲突 | 版本不兼容 | 使用虚拟环境 |
| 导入错误 | 路径问题 | 检查 sys.path |
| 类型错误 | 类型不匹配 | 检查类型注解 |

### Nginx 问题

| 问题类型 | 常见原因 | 解决方案 |
|---------|---------|---------|
| 502 Bad Gateway | 后端服务不可用 | 检查后端状态 |
| 403 Forbidden | 权限问题 | 检查文件权限 |
| 配置语法错误 | 配置项错误 | 检查配置语法 |

## 示例

### 示例 1: Java Spring Boot 问题

**用户提问**: "我的 Spring Boot 应用启动时报错 `BeanCreationException`，怎么解决？"

**AI 回答**:

```markdown
## 问题分析

### 技术栈识别
- **技术大类**: Java
- **技术小类**: Spring Boot
- **组件版本**: 未知（需要确认）
- **操作系统**: 未知（需要确认）

### 问题描述
Spring Boot 应用启动时抛出 `BeanCreationException`，通常是 Bean 创建失败。

---

## 解决方案

### 方案 1: 检查 Bean 依赖关系
**来源**: [Spring 官方文档 - Spring IoC Container](https://docs.spring.io/spring-framework/reference/core/beans.html)
**验证状态**: ✅ 已验证
**可信度**: ⭐⭐⭐⭐⭐ 官方文档

**步骤**:
1. 查看完整的错误堆栈，找到导致失败的具体 Bean
2. 检查该 Bean 的依赖是否正确注入
3. 确认 `@Autowired` 注解的字段是否有对应的 Bean

**验证命令**:
```bash
# 启动应用并查看详细日志
java -jar your-app.jar --debug
```

### 方案 2: 检查组件扫描配置
**来源**: [Spring Boot 官方文档 - Using @SpringBootApplication](https://docs.spring.io/spring-boot/referencesingle/using.html#using.spring-boot-features)
**验证状态**: ✅ 已验证
**可信度**: ⭐⭐⭐⭐⭐ 官方文档

**步骤**:
1. 确认 `@SpringBootApplication` 注解在正确的包路径下
2. 检查组件扫描范围是否覆盖了所有需要的类

## 注意事项

1. **禁止猜测**: 信息不足时必须主动询问用户
2. **强制引用**: 每个解决方案必须附带来源引用
3. **版本验证**: 必须识别并验证组件版本
4. **环境检测**: 自动检测用户工程环境
5. **测试验证**: 提供验证案例和回归测试
6. **未验证标注**: 明确标注未验证信息
7. **持续更新**: 定期更新知识库和规则
